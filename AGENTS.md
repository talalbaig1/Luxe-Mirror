## Cursor Cloud specific instructions

`luxe-mirror` is a single Next.js 16 app (App Router, Turbopack) — web UI + API routes in one process. Commands live in `package.json`: `npm run dev` (port 3000), `npm run build`, `npm run start`, `npm run lint`. The Prisma client is generated by the update script and by `npm run build`; run `npx prisma generate` manually if you edit `prisma/schema.prisma`.

Non-obvious gotchas:
- Middleware lives in `proxy.ts`, not `middleware.ts` (this Next version). It uses Supabase Auth to guard every non-public route, so protected routes (e.g. `/dashboard`, `/onboarding`) redirect to `/sign-in` unless there is a valid Supabase session.
- Auth is **Supabase**, not Clerk — the README is stale (Clerk references are leftover). Env vars actually read by code: `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`, `SUPABASE_SERVICE_ROLE_KEY`, `DATABASE_URL`, `OPENAI_API_KEY`.
- Feature flags (`lib/config.ts`) default to OFF/offline when **unset** (`isDatabaseEnabled` is `true` only if `DATABASE_ENABLED === "true"`). With defaults, the app uses an in-memory store (`lib/db/memory.ts`, resets on restart) and base64 photos — no external Postgres/Storage needed. Set `VISUAL_PREVIEWS_ENABLED=false` to skip AI image generation.
- `.env` is git-ignored. Create one for local dev. Supabase env vars are still required for auth even in offline (in-memory) mode; onboarding/analysis/chat additionally require `OPENAI_API_KEY` (paid).
- Running without cloud credentials: a local Supabase stack (Supabase CLI + Docker, `supabase start`) provides working auth (account sign-up/sign-in). Point `.env` at `http://127.0.0.1:54321` with the printed anon/service keys. Docker/Supabase CLI are dev-time tools and intentionally NOT in the update script; install them ad hoc when you need real auth. The onboarding step still needs a real `OPENAI_API_KEY`.
